## 1. Ingestion Pipeline & Metadata

Ingestion is handled by `EmbeddingService`, `DocumentParserService`, `WikipediaIngestionService`, and `ImageVisionService`. Every source type goes through the same core chunking & dual-store pipeline, but with **different parsing and metadata**.

### 1.1. Chunking Strategy

- **Chunk Size**: 1000 characters with 200-character overlap.
- **Boundary Snapping**: Chunks snap to paragraph breaks (`\n\n`), then sentence boundaries (`. ! ?`), then newlines, then spaces. A minimum chunk size of 40% of `CHUNK_SIZE` prevents tiny fragments.
- **Deduplication**: A SHA-256 hash of the full document text is computed before chunking. If the hash matches an existing document, chunks are **re-pointed** to the new session (metadata update) instead of re-ingesting — saves embedding cost while still giving the new session the relevance boost.

### 1.2. Dual-Store Ingestion

Every chunk is ingested **in parallel** into two stores:
- **Pinecone** (dense vectors): via `VectorStoreService.ingestDocuments()`. Embeddings are generated by Google's embedding model.
- **Postgres** (keyword/BM25): via `saveChunksForKeywordSearch()`. Stored in the `document_chunks` table with a `tsvector` index for full-text search.

### 1.3. Metadata Per Source Type

| Source Type | Metadata Fields | Notes |
|---|---|---|
| **PDF** (`sourceType=PDF`) | `sourceName`, `originalFileName`, `enrichedFileName`, `assetClassification`, `assetTags`, `chunkIndex`, `totalChunks`, `sessionId`, `sourceUrl`, `page`, `charStart`, `charEnd`, `boundingBoxes`, `sectionPath`, `heading` | Richest metadata. `LayoutTextStripper` extracts per-element bounding boxes from PDFBox for exact page-level citation highlighting. Images extracted from PDFs get `imageUrl` + `isImage=true`. |
| **Image** (`sourceType=IMAGE`) | `sourceName`, `originalFileName`, `sourceType`, `chunkIndex`, `sessionId`, `imageUrl`, `isImage=true`, `sourceUrl` | The chunk text is NOT the raw image — it's the **Gemini Vision description** (`SemanticImage.toDenseEmbeddingText()`), which includes summary, OCR text, detected objects, technologies, relationships, and keywords. |
| **Wikipedia** (`sourceType=WIKIPEDIA`) | `sourceName`, `originalFileName`, `sourceType`, `semanticId`, `type` (LEAD_SECTION/PARAGRAPH/IMAGE/WIKITABLE/INFOBOX), `sectionPath`, `chunkIndex`, `sessionId`, `sourceUrl`, `imageUrl` (for wiki images) | Uses **semantic chunking** via `WikipediaIngestionService`. HTML is parsed with Jsoup to extract sections, infoboxes, tables, and images as structurally distinct `SemanticChunk` objects. Categories are fetched from the Wikipedia API. |
| **Text/DOCX/Markdown** | `sourceName`, `originalFileName`, `sourceType`, `chunkIndex`, `totalChunks`, `sessionId`, `sourceUrl` | Minimal metadata. Parsed as plain text with `DocumentParserService.parseTextFile()`. |

### 1.4. Image Vision During PDF Ingestion

When a PDF is parsed, `DocumentParserService.extractAndDescribeImages()` iterates every page, extracts embedded images (minimum 100×100px, max 25 per document), and sends each to `ImageVisionService.describeImage()`. This produces a `SemanticImage` record containing:
- `summary`, `imageType` (CHART/FLOWCHART/DIAGRAM/PHOTO/etc.), `objects`, `technologies`, `relationships`, `ocr` (raw text in image), `keywords`
- The `SemanticImage.toDenseEmbeddingText()` method concatenates all these fields into a keyword-dense string optimized for both semantic and keyword retrieval.

### 1.5. Session Alias System

On ingestion, `EmbeddingService` populates ordinal aliases (`first`, `second`, `third`...) and keyword aliases (filename tokens) into the `SessionUploadState`. These aliases are passed to the LLM Router so it can resolve user references like "the first one" or "the astronomy article" to exact filenames.

## 2. Intent Classification

Intent classification happens in two phases: **Fast Intent** (zero-cost regex) → **LLM Router** (Gemini 3.1 Flash Lite).

### 2.1. Fast Intent Service (`FastIntentService.java`)

A regex/keyword layer that intercepts trivially classifiable queries **before** the LLM router is called. This saves an LLM call entirely.

| Intent | Trigger | Action |
|---|---|---|
| `GREETING_ACK` | "hi", "hello", "hey", "thanks", "good morning", etc. | Bypass retrieval. Return `DirectExecutionPlan` with `Scope.NONE`. Uses `generalPrompt` (no RAG context). |
| `SESSION_INFO` | "what did I upload", "what files", "how many files" | Bypass retrieval. Synthesizes context from `SessionContext.uploadedDocuments()` — no vector search needed. |
| `META_HISTORY` | "what did I ask", "what have we discussed", "earlier" + "said" | Bypass retrieval. Uses conversation history from Postgres as context. |
| `BOT_QA` | "who are you", "what can you do", "your capabilities" | Bypass retrieval. Uses `generalPrompt`. |
| `null` (no match) | Anything complex | Falls through to the **Unified LLM Router**. |

**Why Fast Intent exists**: The LLM Router costs ~200-400ms and consumes tokens. For queries that are obviously greetings or meta-questions, this saves both latency and cost.

### 2.2. Memory Intents (routed by LLM Router, handled in `ContextBuilderService`)

These are classified by the LLM Router (not FastIntent) because they require semantic understanding:

| Intent | Example Query | Data Source | Action |
|---|---|---|---|
| `TIMELINE_QUERY` | "What did we discuss today?" | `ConversationTimelineRepository` (Postgres Layer 3) | Fetches all episodic summaries for this session, formats as chronological timeline. |
| `EPISODIC_SUMMARY` | "Remind me about our past discussion on SpaceX" | `ConversationTimelineRepository` (Postgres Layer 3) | Fetches topic-specific episodic summaries. |
| `SPECIFIC_TOPIC` | "What did you say the distance to the moon was?" | `MessagesPineconeVectorStore` (Pinecone Layer 4) | Runs similarity search against stored User-Assistant pairs to find the specific past answer. |

**Why Memory Intents need the LLM Router**: Unlike greetings, these require understanding whether the user is asking about *documents* vs *past conversations*. "What did we discuss about black holes?" looks like a document query but is actually asking about episodic memory.

## 3. Strategy Classification

When the LLM Router fires, it classifies the query into one of these strategies:

### 3.1. `SINGLE_SOURCE`

- **When**: Default. Standard factual lookup against uploaded documents.
- **Example**: "What is the GDP of India according to the report?"
- **Tier**: Usually `DIRECT` (1 plan). Can be `DECOMPOSE` if comparing entities within one source.
- **Why Needed**: The vast majority of queries are simple factual lookups. This is the hot path optimized for speed and cost.

### 3.2. `META_DOC_SEARCH`

- **When**: User asks *about* their document collection, not *within* a document.
- **Example**: "Which of my uploaded files discuss black holes?"
- **Tier**: Always `DIRECT` (1 plan).
- **Why Needed**: The system prompt for META_DOC_SEARCH instructs the LLM to name specific files and explain *why* they're relevant — using thematic matching, not just keyword matching. Without this, the LLM would try to answer "which files" from a single top-ranked chunk and miss other relevant documents.
- **Special System Prompt**: Tells the LLM to "identify WHICH documents are relevant" and "not require the exact query word to appear verbatim."

### 3.3. `CONCEPT_EXPANSION`

- **When**: Abstract/implicit topic with no obvious keyword match.
- **Example**: "What does the document say about resilience?"
- **Tier**: Always `DECOMPOSE` with `UNION` merge.
- **Plans Generated**: 3-4 concrete, keyword-dense sub-queries expanding the abstract concept (e.g., "resilience" → "overcoming adversity", "mental strength", "recovery from failure", "adaptability").
- **Why Needed**: Embedding models blur abstract concepts. "Resilience" might not match any chunk directly, but "surviving a crisis" or "bouncing back" would. Decomposing into concrete searches and UNION-merging the results gives comprehensive coverage.
- **Special System Prompt**: Tells the LLM to "infer the concept from the underlying evidence" and "map thematic evidence to the abstract concept."

### 3.4. `TIMELINE_QUERY`, `EPISODIC_SUMMARY`, `SPECIFIC_TOPIC`

See [Memory Intents](#22-memory-intents) above.

## 4. Execution Tiers & Plan Classes

The LLM Router outputs both a strategy AND an execution tier. The tier determines the Java class used:

### 4.1. `ExecutionPlan` (Sealed Interface)

```
ExecutionPlan (sealed interface)
├── DirectExecutionPlan   — 1 plan, DIRECT tier
├── StaticExecutionPlan   — N plans, DECOMPOSE tier
└── AdaptiveExecutionPlan — iterative loop, ADAPTIVE tier
```

### 4.2. `DIRECT` → `DirectExecutionPlan`

- **Plans**: Exactly 1 `RetrievalPlan`.
- **Merge**: `NONE`.
- **Flow**: Query → Hybrid Retrieval → Rerank → Generate.
- **Use case**: Simple factual lookup, META_DOC_SEARCH, memory intents.

### 4.3. `DECOMPOSE` → `StaticExecutionPlan`

- **Plans**: Multiple `RetrievalPlan` objects, executed **in parallel** via `Mono.zip()`.
- **Merge**: `UNION` (combine all evidence) or `COMPARE` (group by entity/source).
- **Flow**: Query → N parallel (Hybrid Retrieval → Rerank) → Merge → Generate.
- **Use case**: "Compare X and Y" (COMPARE merge), concept expansion (UNION merge).

### 4.4. `ADAPTIVE` → `AdaptiveExecutionPlan`

- **Plans**: 1 initial plan, plus dynamically generated follow-up plans.
- **Max Iterations**: 2 (configurable via `maxIterations`).
- **Flow**: Query → Retrieval → Observe → LLM decides next action (SEARCH/REFINE/EXPAND/SWITCH_SOURCE/STOP) → Loop or Stop → Generate.
- **Use case**: Complex, goal-oriented queries where the first retrieval may not find everything.
- **Why Needed**: Some queries genuinely need multi-hop reasoning. "What are all the environmental policies mentioned across all my documents?" might need multiple searches with different terms to achieve comprehensive coverage.

### 4.5. `RetrievalPlan` (record)

Every plan carries:
- `id` (purpose string), `optimizedQuery` (LLM-rewritten keyword-dense query), `targetDocuments` (resolved filenames), `executionMode` (RANKED_RETRIEVAL or WHOLE_DOCUMENT), `scope` (CORPUS, SPECIFIC_DOC, or NONE).

## 5. Hybrid Retrieval Pipeline

Implemented in `HybridRetrievalService.java`. This is the core retrieval engine.

### 5.1. Why Hybrid?

- **Dense search** (Pinecone): Catches paraphrases, conceptual matches, and semantic similarity. Misses exact terms, acronyms, product codes, and rare words.
- **Keyword search** (Postgres BM25): Catches exact terms, names, IDs, and rare words. Misses paraphrases and conceptual matches.
- **Fusing both** ensures the reranker always gets the best candidates from both worlds.

### 5.2. Parallel Search

Both searches run in parallel via `Mono.zip()`:
- **Pinecone Dense**: `VectorStoreService.retrieve(query, topK=15, targetDocuments, imageOnly)` — uses Google's embedding model.
- **Postgres BM25**: `DocumentChunkRepository.keywordSearch(query, topK=15)` — native `to_tsvector` / `plainto_tsquery` with `ts_rank`.
- BM25 is **skipped** if the query has fewer than 3 tokens (too short for meaningful keyword matching).

### 5.3. Reciprocal Rank Fusion (RRF)

```
RRF_score(doc) = Σ(1 / (RRF_K + rank))  for each list the doc appears in
```
- `RRF_K = 60` (standard value from the original RRF paper).
- Documents found by **both** retrievers naturally rise to the top since they accumulate scores from both lists.
- **Why RRF over weighted averaging**: BM25 scores and cosine similarity scores live on completely different, unbounded scales. RRF only looks at *rank*, making it scale-free.

### 5.4. Source Diversity

After RRF fusion, `ensureSourceDiversity()` guarantees no single document floods the results:
- First pass: Allow up to **2 chunks per unique `sourceName`**.
- Second pass: Fill remaining slots (up to 15) with the highest-ranked remaining chunks regardless of source.
- This prevents a single long document from dominating all 15 retrieval slots.

### 5.5. Whole-Document Mode

When the execution mode is `WHOLE_DOCUMENT` (user says "summarize this document"), the hybrid pipeline is **bypassed entirely**:
- `HybridRetrievalService.wholeDocumentRetrieve(sourceName)` fetches all chunks of that document from Postgres, ordered by `chunkIndex`, capped at 60 chunks.
- All chunks get a fixed score of 1.0 (no ranking needed — we want the entire document).

## 6. Visual Search Pipeline

Implemented across `RetrievalOrchestrator`, `HybridRetrievalService` (imageOnly mode), and `EvidenceStructuringService`.

### 6.1. When Visual Search Triggers

The LLM Router sets `visual_search: true` when:
- User explicitly asks for images/diagrams/photos.
- Query involves architectures, system designs, data trends, UI layouts, or spatial relationships — even if the word "image" isn't used.

### 6.2. Visual Retrieval Flow

1. `RetrievalOrchestrator` checks `execPlan.isVisualSearch()`.
2. If true, it runs a **parallel** visual pipeline alongside the text pipeline:
   - Calls `HybridRetrievalService.retrieve(..., imageOnly=true)` — this filters both Pinecone and Postgres to only return chunks where `isImage=true`.
   - The retrieved image chunks are **reranked** (`RerankService.rerank()`) with `topK=3`.
   - Then passed through `MultiSignalRanker.rank()`.
   - Top 3 results are converted to `VisualEvidence` objects (containing `imageUrl`, `caption`, `score`, `sourceDocument`).
3. The visual evidence is merged with text evidence in `EvidenceStructuringService.structure()`.

### 6.3. Visual Evidence in Prompt

`EvidenceStructuringService` adds a `Visual Evidence` section at the top of the context block, with each image's display title and detailed caption extracted from the `SemanticImage` description.

## 7. Reranking & Quality Fallback

### 7.1. Cross-Encoder Reranking (`RerankService.java`)

After RRF fusion, the top candidates are rescored by a cross-encoder (Cohere-style API) that takes the `(query, chunk_text)` pair and outputs a precise semantic relevance score. This is more accurate than embedding similarity because it sees both texts together.

### 7.2. Multi-Signal Ranking (`MultiSignalRanker.java`)

Post-rerank heuristics:
- **Session Boost**: Chunks from the current session get a soft score multiplier, prioritizing documents the user is actively working with.
- This is a *boost*, not a filter — a strongly relevant chunk from another session can still outrank a weak match from the current session.

### 7.3. Quality Fallback (Scope Expansion)

In `RetrievalOrchestrator.fallbackQualityCheck()`:
- If the top candidate's score falls below `RetrievalProperties.getPrimaryThreshold()` AND the query was restricted to specific target documents...
- The orchestrator **automatically rewrites** the plan to search `GLOBAL_CORPUS` and retries retrieval.
- This prevents the system from returning "I couldn't find information" when the answer exists in a different document than the user referenced.

## 8. Prompt Assembly & Evidence Structuring

### 8.1. `EvidenceStructuringService.java`

Transforms raw retrieval candidates into a structured, citation-aware context block:
1. **Deduplicates** by chunk ID.
2. **Groups by source document** (`sourceName`).
3. **Merges adjacent chunks** (consecutive `chunkIndex` values from the same source) into coherent passages.
4. **Detects chronological content** (years, timelines) and sorts accordingly.
5. Applies merge operation: `UNION` (combine all), `COMPARE` (group by entity/source), or `NONE`.

### 8.2. `ContextBuilderService.java` — Final Prompt

Assembles the final prompt from:
- **System Prompt**: Either `ragprompt.st` (with `<inferenceDirective>` replaced based on strategy) or `generalPrompt.st` (for greetings/bot QA).
- **Context Block**: The structured evidence from `EvidenceStructuringService`.
- **Session Context**: List of uploaded files, active file name.
- **Conversation History**: Last 10 messages from Postgres, in chronological order.
- **User Question**: The original query.

## 9. Dual-Layer Memory Architecture

### 9.1. Layer 1: Conversation History (Postgres `messages` table)

- Last 10 messages loaded on every query.
- Provides short-term conversational context.

### 9.2. Layer 2: Memory Gating (`MemoryGatingService.java`)

- After generation, checks if the interaction is "memorable" (not a greeting, not a failed answer, not too short).
- If memorable AND `topScore >= 0.4`, the `User: ... \nAssistant: ...` pair is embedded into `MessagesPineconeVectorStore` for long-term retrieval.

### 9.3. Layer 3: Episodic Timeline (`TopicShiftDetectorService.java` → `ConversationTimeline`)

- On every query, `TopicShiftDetectorService` uses an LLM to detect if the topic has shifted.
- If shifted, the closed episode (all messages in that topic thread) is pushed to `memory_jobs` Redis stream.
- `MemoryWorkerService` consumes the stream, uses an LLM to extract: `topic`, `summary`, `entities`, `keywords`, `importanceScore`.
- Saved to Postgres `ConversationTimeline` table (structured, queryable).

### 9.4. Layer 4: Episodic Vectors (Pinecone `EPISODE_SUMMARY`)

- `MemoryWorkerService` also embeds the episodic summary into Pinecone with metadata `type=EPISODE_SUMMARY`.
- Enables semantic search over past conversation episodes.

## 10. LLM Call Inventory

Every LLM call in the system, and where it happens:

| # | LLM Call | Service | Model | Purpose |
|---|---|---|---|---|
| 1 | **Unified LLM Router** | `PlannerService.callUnifiedRouter()` | Gemini 3.1 Flash Lite | Classify strategy, tier, generate plans, extract entities. |
| 2 | **Image Vision (Ingestion)** | `ImageVisionService.describeImage()` | Default ChatClient model | Describe images extracted from PDFs during ingestion. Called per image (max 25/doc). NOT on the query path. |
| 3 | **Topic Shift Detection** | `TopicShiftDetectorService.detectTopicShift()` | Default ChatClient model | Determine if the user's new query is a topic shift. Called once per query. |
| 4 | **Adaptive Loop Controller** | `RetrievalController.decideNextAction()` | Default ChatClient model | Decide next retrieval action (SEARCH/REFINE/EXPAND/STOP). Called once per adaptive iteration. |
| 5 | **Episodic Summarization** | `MemoryWorkerService.executeSummarization()` | Default ChatClient model | Summarize a closed conversation episode. Async background worker, NOT on the query critical path. |
| 6 | **Final Generation** | `LlmWorkerService.streamAnswer()` | User-selected model (via `ModelFactory`) | Generate the final answer. Streamed via Redis Pub/Sub → SSE. |
| 7 | **Suggested Questions (Post-Ingestion)** | `SuggestedQuestionsService.generateForSession()` | Gemini 2.5 Flash Lite | Generate 3 starter questions after document upload. Async, fire-and-forget. |
| 8 | **Follow-Up Questions (Post-Generation)** | `SuggestedQuestionsService.triggerFollowUpForSession()` | Gemini 2.5 Flash Lite | Generate 3 follow-up questions after every assistant response. Async, fire-and-forget. |

## 11. Worst-Case LLM Calls Per Strategy

### 11.1. `GREETING_ACK` / `BOT_QA` / `SESSION_INFO` / `META_HISTORY` (Fast Intent)

| Step | LLM Call? |
|---|---|
| Fast Intent Classification | ❌ (regex) |
| Topic Shift Detection | ✅ (1 call) |
| Generation | ✅ (1 call) |
| Follow-Up Questions | ✅ (1 call, async) |

### 11.2. `SINGLE_SOURCE` / `META_DOC_SEARCH` — DIRECT Tier

| Step | LLM Call? |
|---|---|

| Topic Shift Detection | ✅ (1 call) |
| LLM Router | ✅ (1 call) |
| Hybrid Retrieval | ❌ (vector DB + Postgres) |
| Cross-Encoder Rerank | ❌ (external API, not LLM) |

| Follow-Up Questions | ✅ (1 call, async) |

| Step | LLM Call? |
|---|---|
| Topic Shift Detection | ✅ (1 call) |

| Follow-Up Questions | ✅ (1 call, async) |
| **Total** | **4 calls** (3 on critical path) |

| Class | Purpose |
|---|---|
| `ModelName` (enum) | Available generation models: GEMINI_2_5_FLASH, GEMINI_2_5_FLASH_LITE, GEMINI_2_5_PRO, GEMINI_2_5_PRO_LITE, GEMINI_3_1_FLASH_LITE. Each has `getModelString()` for API calls. |
| `ModelFactory` (service) | Resolves which model to use based on: frontend override → user preference → default. Also injects response style (TECHNICAL/CONCISE/DETAILED/BEGINNER) and language preferences into the system prompt. |
