# Backend Third-Party Services

This document explains the rationale behind each of the major third-party services, libraries, and frameworks used in the **DocuMind** backend.

## Core Framework & Data Storage

*   **Spring Boot 3.5.6 (WebFlux, Data JPA, Security)**
    *   **Why it's used:** The core framework of the backend. Spring Boot provides a robust, enterprise-grade foundation for building REST APIs. We utilize WebFlux for reactive/asynchronous endpoints, Data JPA for easy database interactions, and Spring Security for protecting the endpoints.
*   **PostgreSQL (e.g., Supabase, RDS)**
    *   **Why it's used:** The primary relational database. It stores structured application data such as user accounts, document metadata, file paths, and conversation histories.
*   **Redis**
    *   **Why it's used:** An in-memory data structure store used for high-performance caching. It is critical for quickly retrieving frequently accessed data or managing rate limits without hitting the main PostgreSQL database.

## AI & Machine Learning

*   **Spring AI**
    *   **Why it's used:** Provides a unified abstraction layer over various AI models and vector databases. It prevents hard-coupling the code to specific AI provider APIs, making it easier to swap out LLMs or Vector DBs in the future.
*   **Google Gemini API**
    *   **Why it's used:** The core Large Language Model (LLM) powering DocuMind. It processes the retrieved document chunks, understands user queries, and generates the final, highly accurate natural language answers.
*   **Pinecone (Vector Database)**
    *   **Why it's used:** Specialized database for storing and querying high-dimensional vectors (embeddings). It allows the system to perform ultra-fast "semantic searches"—finding document chunks that mean the same thing as the user's question, rather than just matching exact keywords.
    *   *Note: Two separate accounts are used to strictly isolate conversational session history from the actual document embeddings.*
*   **ONNX Runtime & HuggingFace Tokenizers (`ms-marco-MiniLM-L-6-v2`)**
    *   **Why it's used:** Used to run a local "Cross-Encoder" model. After Pinecone retrieves the initial candidate document chunks, this local ONNX model re-ranks them to determine the absolute most relevant chunks. Running this locally saves API costs and reduces latency compared to sending all chunks to an external re-ranking API.

## File Processing & Utilities

*   **Apache PDFBox**
    *   **Why it's used:** A Java library used to parse uploaded PDF files. It extracts the raw text as well as the geometric `boundingBoxes` (X/Y coordinates) of that text so the frontend can highlight exactly where answers came from.
*   **Cloudinary**
    *   **Why it's used:** A cloud-based media storage service. If the system extracts images from documents or needs to store user-uploaded media, Cloudinary hosts these files and serves them via optimized CDN URLs.
*   **JWT (JSON Web Tokens)**
    *   **Why it's used:** The standard for stateless user authentication. It allows the backend to verify that a user is logged in securely on every request without needing to look up session data in the database.
*   **Jsoup & CommonMark**
    *   **Why it's used:** Utilities for cleaning, parsing, and converting HTML and Markdown. This is crucial for formatting the text extracted from documents or the text generated by the AI before it is sent to the frontend.
